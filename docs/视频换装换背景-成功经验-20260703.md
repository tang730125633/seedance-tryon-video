## 这次最重要的教训

不要把用户给的 4 个视频误认为“1 个人物视频 + 3 个衣服参考”。这次用户明确说的是：

- 3 个 `.mov`
- 1 个 `.mp4`
- 这 4 个都是女生源视频

正确做法是把它们分别放进 4 个客户目录的 `人物视频/` 下面，再单独给每个客户配衣服参考和背景参考。

## RunningHub 主链路

这次稳定可复现的链路仍然是两段式：

1. 先换装
2. 再换背景

### 1. 换装

RunningHub AI App：

- 名称：`视频换装V2 Animate Wan2.2=B站艾橘溪`
- WebApp ID：`1969605116187844610`

节点：

- `363.video`：人物源视频
- `373.image`：衣服参考图
- `362.value`：视频秒数
- `358.value`：宽度
- `359.value`：高度
- `372.text`：替换目标

推荐参数：

- `seconds`: `6`
- `width`: `576`
- `height`: `1024`
- `target`: `Clothes`

关键经验：

- `target` 写 `Clothes` 比写复杂服装描述更稳。
- 复杂中文服装描述容易让模型保留原衣服，或者把领口、肩部结构理解错。
- 衣服参考最好是干净的衣服区域，不要带太多手机 UI、黑边或无关背景。

### 2. 换背景

RunningHub AI App：

- 名称：`VideoRefusion 动态替换视频背景`
- WebApp ID：`1986353521488523266`

节点：

- `352.video`：换装后的视频
- `318.image`：背景图
- `339.int`：视频秒数

关键经验：

- 背景阶段很慢，容易长时间 `RUNNING`，但不一定失败。
- 串行等待会被某一个慢任务卡住，所以建议使用并行续跑脚本。
- 背景阶段可能再次影响脸部和边缘，要做最终质检。

## 推荐生产命令

运行前先设置环境变量，不要把 API key 写进脚本：

```bash
export RUNNINGHUB_API_KEY="你的 RunningHub API Key"
```

串行脚本：

```bash
python3 -u "90-AI脚本与代码/scripts/run_tryon_bg_from_manifest.py" \
  --manifest "换装视频测试/客户素材待处理-指定4视频女生/ready_clients_manifest.json" \
  --out-dir "换装视频测试/批量输出-指定4视频女生换装换背景" \
  --poll-interval 20 \
  --timeout 3600
```

更推荐并行续跑脚本：

```bash
python3 -u "90-AI脚本与代码/scripts/run_tryon_bg_parallel_resume.py" \
  --manifest "换装视频测试/客户素材待处理-指定4视频女生/ready_clients_manifest.json" \
  --out-dir "换装视频测试/批量输出-指定4视频女生换装换背景" \
  --poll-interval 30 \
  --timeout 5400
```

并行续跑脚本的价值：

- 已经提交过的任务不会重复提交。
- 已完成的换装会继续提交背景。
- 某个背景任务长时间 `RUNNING` 时，其他客户不会被堵住。
- 会持续写入 `batch_log.json`，便于恢复和复盘。

## 以后接单的标准动作

1. 先问清楚每个文件的角色：人物视频、衣服参考、背景参考。
2. 不确定时先抽帧做核对图，不要直接跑云端。
3. 每个客户按三件套目录整理。
4. 扫描素材完整性。
5. 生成 manifest。
6. 用并行续跑脚本跑 RunningHub。
7. 生成质检拼图和预览页。
8. 只把经验、脚本、结构上传 GitHub；不要上传客户原视频、API key、任务 signed URL。
