# 工作流原理 · 换装 + 换背景

本文解释这条主链路"为什么这么设计、每一步在做什么"。想直接照抄参数去生产，看 `runninghub-workflow.md`；这里讲原理。

## 一句话

不是"一个模型一步换好"，而是**两段式流水线**：先让 AI 换掉人物身上的衣服，再让另一个 AI 换掉背景。每段都跑在 RunningHub 云端的一个 ComfyUI 工作流（AI App）上，脚本通过 API 上传素材、覆盖几个节点参数、提交任务、轮询等结果、下载成片。

## 整体链路（脚本视角）

```
本地素材 ──上传──> 云端 download_url
      │
      ▼
[第1段] 换装 AI App ──提交──> task_id ──轮询 status──> SUCCESS ──下载──> 换装成片.mp4
      │
      ▼
[第2段] 换背景 AI App ──(换装成片 + 背景图)──> task_id ──轮询──> SUCCESS ──下载──> 最终成片.mp4
      │
      ▼
QA 质检拼图 + 预览页
```

RunningHub 是**异步任务模型**：提交后立刻返回 `task_id`，任务在云端 GPU 上跑；客户端每隔 N 秒（默认 20s）查一次状态（`RUNNING → SUCCESS / FAILED`），成功后再取结果 URL 下载。日志里刷一长串 `status=RUNNING` 就是这个轮询。

## 为什么能"只换衣服/背景，人和动作不变"

### 第 1 段 · 换装 = `视频换装V2 Animate Wan2.2=B站艾橘溪`

- 底层是 **Wan2.2 视频扩散模型**的 `Animate`（动作驱动的角色重绘）变体。
- 原理：把**原视频当作"动作/姿态/镜头的驱动信号"**逐帧提取，然后在这个骨架上**重新绘制出穿着新衣服的同一个人**。
- 因为动作、脸、镜头都由原视频锁定，模型只被要求改**衣服区域的外观**（参考衣服图），所以人物身份和动作能保持稳定——这就是它比"文本一句话生成新视频"稳的根因。
- 由此得出的经验：`372.text` 目标词写 `Clothes` 比写详细服装描述更稳。你只需告诉它"要换的是衣服这个区域"，外观交给参考图决定；描述越复杂，模型越容易保留原衣服或理解错领口/肩部结构。

### 第 2 段 · 换背景 = `VideoRefusion 动态替换视频背景`

- 原理：**逐帧把前景人物分割抠出**，再把背景图重新合成/扩散到人物身后，并做**时间一致性**处理（帧间不闪、不飘）。
- "Refusion"＝把主体重新"融合"进新背景。这一段较慢，且因为要重新处理边缘，**可能再次影响脸和轮廓**，所以成片后必须质检。

## 那些"节点号"是什么

每个 AI App 底层是一张 **ComfyUI 工作流图**（节点连成的管线）。API 不改整张图，只**覆盖某几个节点的输入值**：

换装：
- `363.video`＝原视频 · `373.image`＝衣服图 · `362.value`＝秒数 · `358/359.value`＝宽/高 · `372.text`＝替换目标

换背景：
- `352.video`＝换装成片 · `318.image`＝背景图 · `339.int`＝时长

所以脚本本质就是"把这几个输入插槽填上素材和参数，按运行"。

## 两个工程要点

1. **企业 Key 的并发**：普通消费级 Key 一次只能跑一个任务；企业共享 Key 默认支持 100 并发、按运行时长计费，所以多个客户的换装能同时在云端跑（日志里多个 task 并列 `RUNNING`）。
2. **两段式 + 轮询续跑**：背景阶段常长时间卡 `RUNNING`，串行等待会被某一个慢任务堵住，所以用并行续跑脚本（`run_tryon_bg_parallel_resume.py`）。这也关系到长视频"一次性换背景"能不能成——见 `long-video-boundaries.md`。

## 为什么分两段而不是一步到位

一步同时换装+换背景，模型要同时兼顾"重绘衣服"和"重建背景"，自由度太大、容易崩（保不住人、动作漂移、背景乱）。拆成两段：第 1 段只动衣服、锁住人和动作；第 2 段只动背景、锁住已经换好装的主体。每段目标单一，结果更稳、更可控、也更好质检和续跑。
